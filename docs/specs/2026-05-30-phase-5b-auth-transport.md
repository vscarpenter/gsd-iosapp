# Phase 5b — Auth + Transport (Design Spec)

> **Date:** 2026-05-31 · **Status:** design approved in brainstorming; **NOT yet planned or implemented**. Basis for the implementation plan. Next: write the plan (`docs/superpowers/plans/2026-05-30-phase-5b-auth-transport.md`), then execute subagent-driven. Second slice of **Phase 5 (Sync & Backend)** (5a done; **5b** auth+transport; 5c sync engine; 5d realtime+health+UI).

## 1. Purpose & scope
A complete, **provider-agnostic** authentication + transport layer, wired **Google end-to-end first**. Delivers a `PocketBaseClient` (hand-built `URLSession` REST per §7.0), auth models, a **stateful PKCE** `AuthService` over protocol seams, and the live App edge (`ASWebAuthenticationSession` + Keychain + an Account UI). The output: an authenticated session + a client ready for 5c to make authed CRUD calls.

**Backend ground truth** (captured live from `GET https://api.vinny.io/api/collections/users/auth-methods`, HTTP 200): **PocketBase ≥ 0.23** (modern `oauth2.providers[]` shape), auth collection **`users`**, **Google + GitHub** OAuth2 live (**no Apple**), PKCE (`codeVerifier`/`codeChallenge`/`S256`/`state` per attempt), `authURL` carries an **empty `redirect_uri=`** the client must append. §8.4 identity = **(c) email-keyed** (both providers request email scope).

**Out of scope (deferred):** task CRUD / pull / push / queue-drain / sync-history → **5c**; SSE realtime / safety-net / health → **5c/5d**; **GitHub + Apple wiring** → later config (the code is provider-agnostic, so it's config not rework); the 5a `decodeList` pagination unwrap (`{items:[…]}`) → **5c** (list responses, not auth).

## 2. Builds on (Phases 0–5a, on `main` at `phase-5a-sync-foundation`)
- **`GSDSync`** (Foundation-only, depends `GSDModel`): the 5a units (`PocketBaseTaskRecord`, `WireDate`, `TaskWireMapper`, `LWW`) — all internal. 5b adds the client + auth here; it **stays Foundation-only** (no `GSDStore` dep yet — that's 5c) and gains **no** new framework imports (ASWebAuthenticationSession + Keychain live in the App behind protocol seams).
- **`GSDStore`**: `DeviceIdentity` + `SyncQueue` (used in 5c, not 5b).
- **App**: `GSDApp` DI wiring (`@State store`, injected deps — the pattern 5b extends), `SettingsView` (Appearance/Archive/Data&Storage/Notifications/About — 5b adds an **Account** section), the `ReminderScheduling` → `LiveReminderScheduler` **protocol-seam precedent** (5b mirrors it: `WebAuthPresenting`/`TokenStore` protocols in the package, live impls in the App).
- **Package/project**: `project.yml` (XcodeGen source of truth) — 5b adds the `gsd` URL scheme (`CFBundleURLTypes`) for the OAuth callback.

## 3. Scope calls (design — approved in brainstorming)
- **`PocketBaseClient`** (GSDSync, Foundation/`URLSession`): config base URL; **auth methods for 5b** — `authMethods() -> AuthMethods` (GET `/api/collections/users/auth-methods` — **unauthenticated**; the auth-header injection is per-request and omitted for this bootstrap call, confirmed by the live curl), `authWithOAuth2(provider:code:codeVerifier:redirectURL:) -> AuthResult` (POST `/api/collections/users/auth-with-oauth2`), `authRefresh(token:) -> AuthResult` (POST `/api/collections/users/auth-refresh`); plus a **generic authed-request helper** (5c consumes it for CRUD). Typed error mapping (HTTP status + PocketBase error body → `PocketBaseError`). Tested via an **injected request-executor seam** (`(URLRequest) async throws -> (Data, HTTPURLResponse)`) so fixtures drive responses without a network/`URLProtocol`.
- **Auth models** (GSDSync, decode the modern `oauth2.providers[]`): `AuthMethods {providers: [OAuthProvider]}`, `OAuthProvider {name, displayName, state, authURL, codeVerifier, codeChallenge, codeChallengeMethod}`, `AuthResult {token: String, record: AuthRecord}` where `AuthRecord {id, email, …}` (enough to surface the account). Defensive decode (5a precedent).
- **`AuthService`** (GSDSync, the **stateful** PKCE orchestration over seams): `signIn(provider:) async throws -> Session` — fetch methods → hold the provider's `{state, codeVerifier}` → append `redirect_uri` to `authURL` → `WebAuthPresenting.present(...)` → **validate returned `state` == held `state`** (CSRF; else `stateMismatch`) → extract `code` → `authWithOAuth2(...)` → `TokenStore.save(token)` → return session. **Invariant: exactly one `auth-methods` fetch per `signIn`** — the `{state, codeVerifier}` are per-attempt and ephemeral; never cache `AuthMethods` across attempts (a reused `codeVerifier` fails the provider's PKCE challenge **live**, and the fake presenter won't catch it — A51 asserts the threaded verifier). Plus `signOut()` (clear token), `refresh()` (extend valid JWT), `validToken() async -> String?` (proactive refresh before `exp`). **No refresh-token model** — on expiry, clear + require re-auth.
- **Seams** (GSDSync protocols, `Sendable`): `WebAuthPresenting { present(authURL: URL, callbackURLScheme: String) async throws -> URL }`; `TokenStore { load() -> String?; save(_:); clear() }`. The App provides live impls; tests use a fake presenter (canned callback URL) + in-memory store.
- **App live edge**: `LiveWebAuthPresenter` (`ASWebAuthenticationSession` + `ASWebAuthenticationPresentationContextProviding` anchor) — **distinguishes user-cancel** (completion `canceledLogin` → `cancelled`: silent return to signed-out, no error banner) **from presentation failure** (`start()` returns false / missing anchor → a surfaced `presentationFailed` error, NOT swallowed as a cancel); `KeychainTokenStore` (`Security`) — a **plain Keychain item, NO access group, in 5b** (the shared access group of §8.3 needs a team/app-ID-prefix entitlement + an actual extension, both of which arrive in **Phase 6**; adding it now collides with the no-`DEVELOPMENT_TEAM`-in-commits rule and has no consumer yet); `SessionStore` (`@MainActor @Observable` app-facing state: `signedIn`/`account email`/`inProgress`/`error` — wraps `AuthService`, keeps auth UI-state **out** of the pure orchestration, per the `TaskStore` precedent); an **Account** section in `SettingsView` (provider sign-in button(s) — Google for 5b; signed-in shows email + Sign Out); `GSDApp` wiring (construct client + live seams + `AuthService` + `SessionStore`; inject).
- **Redirect config** (injected): callback scheme `gsd://oauth-callback` (registered in `project.yml` `CFBundleURLTypes`); redirect URI `https://api.vinny.io/ios-oauth-redirect/` (a backend bounce page → `gsd://` — user-provisioned). The **same redirect URI** is used to build `authURL` AND in `authWithOAuth2` (Google requires exact match).
- **Provider-agnostic**: the flow is identical per provider (only the `provider` name + scopes differ); Google is wired/tested first, GitHub/Apple are later config.

## 4. Feature groups (→ plan groups)
- **A — `PocketBaseClient` + auth models (GSDSync, `swift test`):** the client (request building, response/error decode) + `AuthMethods`/`OAuthProvider`/`AuthResult`; fixtures authored from the **captured** `auth-methods` shape + representative `auth-with-oauth2`/error bodies. Probe (in `/tmp`): JWT `exp` decode + `authURL` redirect-append/encoding.
- **B — `AuthService` PKCE orchestration (GSDSync, `swift test`):** the full flow over a **fake `WebAuthPresenting`** + **in-memory `TokenStore`** — happy path, **`state` mismatch** rejection, user-cancel (non-error), token persisted; `refresh()`/`validToken()` (exp decode, proactive refresh, re-auth on 401/expiry).
- **C — Live edge + Account UI (App, build + MANUAL):** `LiveWebAuthPresenter`, `KeychainTokenStore`, `SessionStore`, the Settings **Account** section, `GSDApp` wiring, `project.yml` `gsd` URL scheme (→ `xcodegen generate`). Runtime-verified.

## 5. Testing
- **`swift test` (GSDSyncTests):** client decode of the real `auth-methods` (`oauth2.providers[]`), `auth-with-oauth2` `AuthResult`, and PocketBase error bodies → typed errors; `AuthService` over fake seams (happy / stateMismatch / cancel / persisted) + refresh logic. Fixtures are **grounded in the captured response** (not spec guesses).
- **Build + MANUAL (the §8 runtime edge — NOT unit-testable):** the real `ASWebAuthenticationSession` Google round-trip on a sim/device; Keychain persistence across launches; the Account UI render + sign-in/out.
- **Definition of Done (the deliberate break from 5a):** unit-green is necessary but **NOT sufficient** — a passing fixture suite proves client/orchestration *self-consistency, not that sign-in works*. DoD **requires the live Google round-trip** against `api.vinny.io` (a real token obtained + persisted across launch) **and** empirical confirmation of §8.4(c) email convergence. **5b is NOT merged to `main` on unit-green alone** — it stays on-branch (the Account UI behind a not-yet-shipped surface) until the live flow is validated. Needs: the user's Google redirect setup (§3) + a test login.

## 6. Acceptance criteria (A49–A56)
- **A49** `PocketBaseClient` decodes the real `auth-methods` (`oauth2.providers[]` → `OAuthProvider` incl. `state`/`codeVerifier`/`codeChallenge`); maps HTTP status + PB error body → typed `PocketBaseError`. Unit-tested against captured fixtures.
- **A50** `authWithOAuth2` (POST → `AuthResult {token, record}`) and `authRefresh` build correct requests and decode results; the request-executor seam lets tests drive responses. Unit-tested.
- **A51** `AuthService.signIn` runs the stateful PKCE flow: fetch methods → hold `{state, codeVerifier}` → append redirect → present (seam) → exchange → store token. The test **asserts the `codeVerifier` sent to `authWithOAuth2` is the one minted by THIS `signIn`'s `auth-methods` fetch** (guards the no-cache invariant — a stale verifier would fail PKCE live but pass the fake presenter). Unit-tested via a fake presenter + in-memory store.
- **A52** `signIn` **validates the returned `state`** (CSRF) → `stateMismatch` on mismatch; user-cancel → `cancelled` (non-error; `SessionStore` returns to signed-out with no error banner), distinct from a presentation failure (a surfaced error). Unit-tested.
- **A53** Token refresh **extends a valid JWT** (`exp` decoded, proactive refresh via `validToken()`); on expiry/401 the session clears and re-auth is required (no refresh-token). Unit-tested.
- **A54** `LiveWebAuthPresenter` (`ASWebAuthenticationSession`) + `KeychainTokenStore` (`Security`, access group) build and conform to the seams. Build-verified; behavior is manual.
- **A55** `SessionStore` (`@MainActor @Observable`) drives a Settings **Account** section (Google sign-in, signed-in email, Sign Out); `gsd` URL scheme registered. Build + simctl-render verified; flow is manual.
- **A56** **Live round-trip (manual, gates merge):** real Google sign-in via `ASWebAuthenticationSession` against `api.vinny.io` returns a token, persisted across launches; §8.4(c) email convergence confirmed (same email → same PocketBase user as web). **Not merged to `main` until this passes.**
- **Coverage:** `swift test` green for A+B (GSDSync); both sims build; the live round-trip + Keychain + email-convergence done as the manual gate.

## 7. Conventions (carried + new)
`GSDSync` stays **Foundation-only** and **`GSDModel`-only** (NO `GSDStore` until 5c; NO `AuthenticationServices`/`Security` in the package — those live in the App behind `WebAuthPresenting`/`TokenStore`, mirroring the Phase-4 `ReminderScheduling` seam). App imports `AuthenticationServices`/`Security`/`SwiftUI`. `SessionStore` is `@MainActor @Observable` (TaskStore precedent); auth UI-state never leaks into the pure `AuthService`. Inject config (base URL, redirect URI, callback scheme) + the request-executor + the seams for deterministic tests. **Fixtures from REAL captured responses** (the 5a `decodeList` lesson — auth is where a spec guess is costliest). `GSDModel.Task` shadows Swift's `Task` → `_Concurrency.Task`. Swift Testing for logic; `xcodebuild`+simctl for the App; **`xcodegen generate`** after the `project.yml` URL-scheme change; no `DEVELOPMENT_TEAM` in commits. Probe JWT-`exp` decode + `authURL` building in `/tmp` before the plan.

## 8. Watch-outs
- **Stateful PKCE (not a one-shot exchange):** `AuthService` must hold the per-attempt `{state, codeVerifier, redirectURL}` from `auth-methods` through the exchange; **validate `state`** (CSRF); the **redirect URI must match exactly** between the auth request and `authWithOAuth2` (Google enforces it). **Never cache `AuthMethods`/`codeVerifier` across attempts** — exactly one fetch per `signIn` (a stale verifier fails PKCE live; the fake presenter masks it, so A51 asserts the threaded verifier).
- **Token refresh = extend a still-valid JWT** — PocketBase has no long-lived refresh token; once expired, re-run OAuth. Refresh proactively before `exp`.
- **The shared Google provider:** PocketBase has ONE `users` Google provider, used by the **web app too** — do NOT reconfigure it to an iOS client (breaks web). Native reuses the existing web client + an added `https` redirect URI + a bounce page → `gsd://` (user-provisioned, §3). The client treats the redirect URI as config.
- **Don't carry 5a's merge bar:** auth fixtures prove self-consistency, not working sign-in → DoD requires the **live round-trip**; stay on-branch until validated (A56).
- **`ASWebAuthenticationSession` + Keychain are runtime** (build + manual); the protocol seams keep `GSDSync` deterministic in `swift test` (no real browser/Keychain in tests).
- **Reconcile at live-wiring:** the `auth-with-oauth2` success + error body shapes in fixtures are best-effort until a real token exchange is observed; capture a real exchange response during the manual round-trip and reconcile (same discipline as the 5a fixture-reconcile). **Specifically confirm whether `auth-with-oauth2` requires `state` in the POST body** (PocketBase may validate it server-side by version) — omitting it passes every fixture but could fail live.
- **§8.4(c) is verified live, not in code:** the iOS client is identity-agnostic; confirm convergence empirically (sign in with the email used on web → land on the same PocketBase user).
